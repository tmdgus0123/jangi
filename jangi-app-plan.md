# 장기 앱 개발 기획 및 진행 계획

작성일: 2026-04-21

## 1) 목표 정의

- 모바일과 웹에서 모두 플레이 가능한 장기(한국 장기) 앱 개발
- 1차 목표는 "규칙이 정확한 싱글 플레이 + 로컬 2인 + 온라인 2인" 제공
- 2차 목표는 랭킹/전적, AI 난이도, 리플레이 분석 기능 확장

핵심 성공 기준:

- 불법 수(규칙 위반 수)가 100% 차단될 것
- 턴 전환/체크 상태/승패 판정 로직이 안정적으로 동작할 것
- 모바일 기준 60fps에 가까운 체감 성능 유지

## 2) 권장 개발 환경

### 운영/도구

- OS: macOS (현재 환경)
- IDE: VS Code
- 버전 관리: Git + GitHub
- 패키지 매니저: pnpm (모노레포 운영 시 유리)
- 런타임: Node.js 22 LTS

### 품질/협업

- TypeScript strict 모드 활성화
- ESLint + Prettier + Husky + lint-staged
- 테스트: Vitest + Playwright
- CI: GitHub Actions (lint/test/build 자동화)

## 3) 언어/프레임워크 제안

### 기본 선택 (추천)

- 프론트엔드: React + TypeScript + Vite
- 상태 관리: Zustand
- 실시간 통신: Socket.IO
- 백엔드: Node.js + NestJS (TypeScript)
- DB: PostgreSQL + Prisma
- 캐시/매칭 대기열: Redis
- 배포:
  - 웹: Vercel(프론트), Render/Fly.io(백엔드)
  - DB/Redis: Supabase + Upstash 또는 관리형 서비스

추천 이유:

- TypeScript 단일 언어로 프론트/백엔드 생산성 및 안정성 확보
- 게임 규칙 엔진을 공용 패키지로 분리해 재사용 가능
- 실시간 대전 기능(Socket.IO) 구현 속도가 빠름

### 대안

- Flutter + Dart(모바일 우선) + Firebase(빠른 MVP)
- Unity(C#) (애니메이션/효과 중심의 게임성 강화 시)

## 4) 구현 방식(아키텍처)

모노레포 구조 권장:

- apps/web: React 웹 앱
- apps/server: NestJS API + Socket 서버
- packages/game-engine: 장기 규칙 엔진(순수 함수)
- packages/shared-types: DTO/타입 공유

핵심 원칙:

- 게임 규칙 엔진은 UI/네트워크와 완전히 분리
- 모든 수는 서버에서 최종 검증(치트 방지)
- 클라이언트는 낙관적 업데이트 가능, 서버 불일치 시 롤백

### 도메인 모델 초안

- Piece: 종류(차/포/마/상/사/장/졸), 소속, 위치
- Board: 9x10 좌표 상태
- GameState:
  - 현재 턴
  - 기물 배치
  - 수 기록(move history)
  - 체크/승패 상태
- MoveValidation:
  - 말별 이동 규칙
  - 궁성/포/장군 관련 규칙
  - 자가 장군 금지

## 5) MVP 범위

필수 기능:

- 대국 생성/참가
- 말 선택/이동 및 규칙 검증
- 턴 진행, 기보 저장
- 승패 처리(장군/외통/기권/시간패 중 우선 구현 범위 선택)
- 로컬 2인 모드
- 온라인 랜덤 매칭 또는 방 코드 입장

후순위 기능:

- 관전 모드
- 리플레이
- AI 대전(기초 휴리스틱)
- 랭킹/시즌

## 6) 단계별 진행 계획

### Phase 0. 요구사항 정리 (2~3일)

- 장기 규칙 상세 명세 문서화
- MVP 포함/제외 범위 확정
- 와이어프레임(대국판, 대기실, 결과 화면)

산출물:

- 규칙 명세서
- 화면 목록 및 사용자 플로우

### Phase 1. 프로젝트 부트스트랩 (1~2일)

- 모노레포 초기화
- 공통 린트/포맷/테스트 환경 구성
- CI 파이프라인 구성

산출물:

- 빌드 가능한 초기 코드베이스
- PR 체크 자동화

### Phase 2. 규칙 엔진 개발 (5~8일)

- 보드/기물 모델 정의
- 이동 생성기 및 유효성 검증 구현
- 체크/체크메이트(외통) 판정
- 단위 테스트 고도화

완료 기준:

- 핵심 규칙 테스트 케이스 통과율 100%

### Phase 3. 싱글/로컬 대전 UI (4~6일)

- 대국판 렌더링
- 드래그/탭 이동 인터랙션
- 기보 패널/되돌리기(선택)

완료 기준:

- 브라우저에서 로컬 2인 대전 정상 진행

### Phase 4. 온라인 대전 서버 (6~9일)

- 소켓 룸/세션/재접속 처리
- 서버 권위 검증 및 동기화
- 타이머, 기권, 종료 처리

완료 기준:

- 서로 다른 클라이언트 2개에서 안정적 실시간 대전

### Phase 5. 운영 준비 (3~5일)

- 로깅/모니터링(Sentry 등)
- 배포 자동화
- 성능 점검 및 버그 픽스

완료 기준:

- 외부 사용자 대상 베타 배포 가능

- 초반 범위 과대

- 모노레포 루트 설정 완료
- 웹/서버/공용 타입/규칙 엔진 패키지 초기 뼈대 생성 완료
- 기본 CI 워크플로 추가 완료
- 규칙 엔진 초기 상태 생성 함수 및 샘플 테스트 추가 완료
- 규칙 엔진 전체 시작 배치 초안 및 좌표 조회 헬퍼 추가 완료
- 기물별 기본 이동 규칙 및 이동 적용 API 초안 추가 완료
- 체크 판정과 자가 장군 방지 기반의 합법 수 필터 추가 완료
- 웹 대국판에 말 선택, 이동 가능 칸 하이라이트, 수 적용, 턴 전환 인터랙션 추가 완료
- 기보 로그, 마지막 수 표시, 체크메이트/종료 판정, UI 승패 표시 추가 완료
- 체크 대상 장 강조, 종료 결과 오버레이, 엔진 레벨 불법 수 차단 추가 완료
- 체크메이트 데모 시나리오(5,6 차 -> 5,4) 스모크 테스트 완료

다음 작업 우선순위:

- 기보 로그를 서버 authoritative 이벤트 구조와 연결
- 체크/체크메이트 알림 연출 및 대국 결과 화면 분리
- 서버 authoritative 동기화 모델 설계

## 12) 데모 실행 가능 기준

현재 상태로 가능한 데모:

- 웹에서 로컬 2인 수동 대전
- 말 선택, 합법 수 하이라이트, 수 적용
- 체크/체크메이트 판정
- 마지막 수 및 기보 로그 표시
- 종료 시 승패 표시 및 재시작
- 체크메이트 재현 시나리오를 버튼으로 로드하고 1수로 종료 검증 가능

데모를 실제로 실행 가능한 수준으로 마무리하려면 남은 핵심 작업:

- `corepack + pnpm 10.x` 기준으로 의존성 설치와 로컬 실행 검증 완료
- 실제 브라우저에서 대국 1판 기준 스모크 테스트 완료
- 체크메이트가 안정적으로 재현되는 최소 시나리오 검증
- README에 데모 실행 명령과 전제 조건 명시

외부 시연용 데모까지 보려면 추가로 필요한 작업:

- 보드 하이라이트/결과 화면 UX 정리
- 오류 없는 빌드 확인
- 간단한 배포 환경 연결 또는 로컬 시연 절차 고정

## 10) 의사결정 포인트

아래 항목이 정해지면 구현 속도가 크게 빨라집니다.

- 첫 릴리스 타깃: 웹 우선 vs 모바일 우선
- 온라인 매칭: 랜덤 자동 매칭 vs 친구 초대 코드
- AI 기능 포함 여부: MVP 포함 vs 추후 릴리스
- 인증 방식: 게스트 우선 vs 소셜 로그인 포함
